# TradePilot V2 第一阶段阶段设计：统一数据接入框架

## Context

本阶段不是直接实现某一个采集器，而是先把 **统一数据接入框架** 设计清楚，并与总设计文档 `docs/V2-投资工作流设计.md` 对齐，确保后续可以按阶段持续推进。

当前 V1 已有后端骨架、DuckDB、分析模块和 API，但数据层仍主要围绕 MockProvider 组织，且 `tradepilot/api/market.py` 直接依赖 `MockProvider`。如果不先统一不同来源的数据接入方式，后续接入 AKShare、B 站视频、实时资讯时会各自长出不同入口、存储方式和处理链路，最终难以复用、难以调试、也难以扩展到筛选层与分析层。

本设计的目标是：
1. 先定义统一的“采集源 -> 原始落库 -> 处理状态 -> 标准化输出 -> 手动/定时触发”框架。
2. 第一批优先支持三类来源：**AKShare 行情/板块数据、B 站视频字幕内容、实时资讯/公告**。
3. 产出一份便于继续执行的阶段设计文档，后续可据此逐步实现，不必重新梳理整体架构。

## Recommended Approach

### 1. 阶段范围

本阶段聚焦“第一层采集层 + 接入框架”设计，不展开完整筛选层/调研层实现。

**本阶段纳入：**
- 统一数据源接入抽象
- Provider 与 Collector 的职责分层
- 原始数据落库模型与处理状态模型
- 手动触发 API 与后续调度器预留接口
- AKShare / B 站视频 / 实时资讯 三类来源的接入边界
- 与现有分析层、前端 API 的衔接方式

**本阶段不纳入：**
- LLM 提取细节实现
- 统一信号总线完整实现
- 每日扫描/简报/预警完整实现
- 前端页面完整交互细节
- APScheduler 具体作业编排实现

### 2. 总体分解：Provider / Collector / Ingestion Service 三层

建议把接入框架拆成三层，而不是把所有来源都塞进 `DataProvider`。

#### 2.1 Provider 层：面向“结构化行情数据”
适用于 AKShare 这一类可直接返回结构化表格数据的来源。

职责：
- 提供统一读取接口
- 返回标准 DataFrame
- 只关心“取数”，不负责落库、状态管理、处理流程

基于现有 `tradepilot/data/provider.py` 扩展：
- 保留现有 7 个方法
- 新增 4 个方法：
  - `get_stock_weekly()`
  - `get_stock_monthly()`
  - `get_sector_stocks()`
  - `get_stock_sector()`

建议新增工厂入口：
- `tradepilot/data/__init__.py` 提供 `get_provider()`
- API 和后续服务统一通过工厂拿 provider，不再直接 new `MockProvider`

#### 2.2 Collector 层：面向“非结构化或半结构化外部来源”
适用于新闻、公告、B 站视频这类来源。

职责：
- 拉取原始内容
- 生成来源级唯一标识
- 写入 DuckDB 原始表
- 维护处理状态（未处理 / 已提取 / 失败 / 跳过）
- 暴露统一的 `sync()` 或 `collect()` 入口

建议目录：
- `tradepilot/collector/news.py`
- `tradepilot/collector/bilibili.py`
- 后续预留 `tradepilot/collector/community.py`

#### 2.3 Ingestion Service 层：统一编排入口
新增一个统一编排层，负责把 Provider 与 Collector 组织成一致流程。

建议目录：
- `tradepilot/ingestion/service.py`
- `tradepilot/ingestion/models.py`
- `tradepilot/ingestion/types.py`

职责：
- 统一定义 source 类型（market/news/bilibili）
- 统一手动触发入口
- 统一记录一次 sync 的开始/结束/结果统计
- 封装“增量更新”策略
- 为未来调度器直接复用

### 3. 统一生命周期设计

所有来源统一走以下生命周期：

1. `discover`：发现要抓取的对象
   - AKShare：股票/指数/板块/日期区间
   - 新闻：资讯流、公告流
   - B 站：频道 / 播放列表 / 视频 URL
2. `fetch`：拉取原始数据
3. `normalize`：标准化字段命名与基础元数据
4. `persist_raw`：写入原始表或行情表
5. `mark_state`：记录处理状态
6. `emit_for_processing`：供后续筛选层消费（本阶段先只预留）

这里最关键的是：**原始采集成功** 与 **后续已处理成信号** 要分离，不能混成一个步骤。

### 4. 数据模型与持久化设计

#### 4.1 复用现有行情表
复用 `tradepilot/db.py` 里已有 7 张结构化表：
- `stock_daily`
- `index_daily`
- `etf_flow`
- `margin_data`
- `northbound_flow`
- `stock_valuation`
- `sector_data`

并扩展以下表：
- `stock_weekly`
- `stock_monthly`
- `sector_stocks`
- `stock_sector_map`

用途：
- 周/月线不强依赖实时计算，优先落库存储，便于技术分析复用
- 板块成分与反向映射单独建表，便于 sector rotation、watchlist 和高切低逻辑复用

#### 4.2 原始采集表
参考总文档第七节，但建议在“processed BOOLEAN”基础上升级为更明确的状态字段：

建议原始表：
- `news_items`
- `video_content`

建议统一补充字段：
- `source`
- `source_item_id`
- `content_hash`
- `collected_at`
- `processing_status` (`pending` / `processed` / `failed` / `skipped`)
- `processing_error`
- `processed_at`

这样后续筛选层能明确区分：
- 尚未处理
- 已成功提取
- 处理失败待重试
- 命中去重规则而跳过

#### 4.3 同步执行历史表
建议优先引入总文档中的 `scheduler_history` 思路，但将语义扩展为更通用的 ingestion 运行历史。

建议表名可保持：`scheduler_history`
也可在实现时重命名为：`ingestion_runs`

记录字段建议包含：
- `job_name`
- `source_type`
- `started_at`
- `finished_at`
- `status`
- `records_discovered`
- `records_inserted`
- `records_updated`
- `records_failed`
- `error_message`
- `trigger_mode` (`manual` / `scheduled`)

### 5. 三类优先来源的接入设计

#### 5.1 AKShare
定位：结构化 provider + 增量入库主来源。

设计重点：
- `AKShareProvider` 实现 `DataProvider`
- 不让 API 直接访问 AKShare，而是统一通过 provider factory
- 由 ingestion service 负责调用 provider 并写入 DuckDB
- 增量逻辑基于各表最大日期
- 降级逻辑：AKShare 限流/失败时返回明确失败状态；是否自动回退 Mock 作为配置项，而不是硬编码默认行为

建议后续修改路径：
- `tradepilot/data/provider.py`
- `tradepilot/data/__init__.py`
- `tradepilot/data/akshare_provider.py`
- `tradepilot/api/market.py`
- `tradepilot/db.py`
- `tradepilot/config.py`

#### 5.2 实时资讯 / 公告
定位：原始资讯采集入口，先落原始库，后续再进入 LLM 提取。

设计重点：
- 第一批范围收敛为 **快讯 + 公告**
- 统一抽象为 `NewsCollector`
- 资讯流与公告流共享一套原始模型，但保留 `source`、`category`、`published_at`
- 去重优先使用 `source + source_item_id`，其次退化到 `content_hash`
- 本阶段先支持“手动同步 + 原始落库 + 状态记录”
- 后续筛选层从 `processing_status = pending` 的记录中取数据处理

建议后续修改路径：
- `tradepilot/collector/news.py`
- `tradepilot/api/collector.py`（新增）
- `tradepilot/db.py`
- `tradepilot/config.py`

#### 5.3 B 站视频 / 字幕
定位：长文本内容来源，和新闻不同，重点在“外部工具链整合 + 文件系统引用 + 原文溯源”。

设计重点：
- 第一批入口按 **UP 主动态 + 投稿** 设计，而不是只围绕单视频 URL
- 复用现有字幕工具链，而不是重写下载与转码能力
- Collector 负责：发现 UP 主新投稿/动态关联视频 -> 下载字幕 -> 转 markdown / 文本 -> 元信息入库
- DB 存元信息与文件路径，长文本正文仍落文件系统
- 统一纳入 `processing_status` 生命周期
- 支持按 UP 主维度手动同步，并为后续补充单视频/合集入口预留参数结构

建议后续修改路径：
- `tradepilot/collector/bilibili.py`
- `tradepilot/api/collector.py`（新增）
- `tradepilot/db.py`
- 相关字幕工具封装入口（以 Explore 阶段找到的现有工具路径为准）

### 6. API 与调用链设计

建议新增统一采集路由，而不是把同步行为继续混入 market API。

新增：
- `tradepilot/api/collector.py`

建议端点：
- `POST /api/collector/market/sync`
- `POST /api/collector/news/sync`
- `POST /api/collector/bilibili/sync`
- `GET /api/collector/runs`
- `GET /api/collector/status`

现有 `tradepilot/api/market.py` 的定位调整为：
- 只负责读数据
- 不直接 new `MockProvider`
- 列表接口可继续保留，但来源改为 provider factory 或数据库读取

调用链建议：
- API Router -> Ingestion Service -> Provider/Collector -> DB
- Analysis Router -> DB / Provider Factory

### 7. 与现有骨架的衔接点

建议重点复用以下现有文件与模式：

- `tradepilot/data/provider.py:5`
  - 现有结构化数据抽象起点，继续沿用而不是另起一套 market adapter
- `tradepilot/data/mock_provider.py:36`
  - 作为 AKShareProvider 的行为参照与本地回退源
- `tradepilot/db.py:17`
  - 当前所有 DDL 都集中在这里，阶段一继续沿用同一初始化入口
- `tradepilot/config.py:3`
  - 当前配置非常薄，适合扩展 provider 类型、AKShare 开关、B 站存储路径、资讯采集参数
- `tradepilot/main.py:5`
  - 当前 FastAPI 装配简单，后续可直接扩展 collector router 与 lifespan
- `tradepilot/api/market.py:1`
  - 当前暴露了直接读取 market 数据的 API，但强依赖 MockProvider，需要作为“去硬编码 mock”的第一落点

### 8. 建议的执行顺序

#### Step 1：先完成文档与骨架统一
- 固化本阶段设计文档
- 明确 Provider / Collector / Ingestion Service 三层边界
- 明确三类优先来源的统一状态模型

#### Step 2：打通结构化入口（AKShare）
- 加 `get_provider()` 工厂
- API 去掉对 `MockProvider` 的直接依赖
- `AKShareProvider` + DuckDB 增量写入跑通
- 补齐周/月线与板块映射表

#### Step 3：打通非结构化原始入口（实时资讯）
- 建 `NewsCollector`
- 跑通原始资讯入库、去重、状态字段
- 提供手动 sync API

#### Step 4：打通长文本入口（B 站）
- 封装现有字幕工具链
- 打通视频元信息入库 + 文件路径引用
- 提供单视频 / 频道同步手动入口

#### Step 5：统一运行记录与后续处理预留
- 接入运行历史表
- 为后续 LLM 提取层预留未处理数据读取接口
- 为 scheduler 接入预留统一 job 入口

### 10. Verification

后续实现时按以下顺序验证：

1. **数据库初始化验证**
   - 启动后 DuckDB 自动创建新增表
   - 检查结构化表、原始表、运行历史表均存在

2. **AKShare 打通验证**
   - 手动执行 market sync
   - 验证 `stock_daily` / `stock_weekly` / `stock_monthly` / `sector_stocks` / `stock_sector_map` 入库
   - 验证 `/api/market/*` 不再硬编码 `MockProvider`

3. **实时资讯打通验证**
   - 手动执行 news sync
   - 验证 `news_items` 正确去重、写入、状态初始化为 `pending`
   - 重复执行时新增与跳过统计正确

4. **B 站视频打通验证**
   - 通过单视频 URL 执行一次 sync
   - 验证字幕文件产出、路径入库、元信息完整
   - 验证失败记录进入 `failed` 状态而不是静默丢失

5. **统一运行历史验证**
   - 每次 sync 后都有 run 记录
   - 状态、计数、错误信息可通过 API 查看

6. **端到端联通验证**
   - API 可手动触发三类来源同步
   - 同步结果可在 DB 中追踪
   - 为后续筛选层保留明确的未处理数据读取入口

## Notes for Execution

实现时优先避免两种问题：
1. 把所有来源都强行塞进 `DataProvider`，导致 B 站/资讯这类非结构化来源语义失真。
2. 让 Collector 直接承担 LLM 提取与分析职责，导致采集层与筛选层边界混乱。

推荐的总原则是：
- **结构化来源走 Provider**
- **内容型来源走 Collector**
- **统一由 Ingestion Service 编排、统一状态、统一触发入口**
