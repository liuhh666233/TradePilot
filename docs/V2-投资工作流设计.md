# TradePilot V2 — A股投资工作流系统设计

> 本文档已同步当前仓库实现进度。除中长期规划外，文中会明确标注“已实现 / 当前版本 / 待扩展”，用于同时承载设计与落地状态。

## 一、项目现状

### 1.1 当前能力快照 (已同步到当前实现)

| 维度 | 状态 | 说明 |
|------|------|------|
| 后端骨架 | 完整 | FastAPI + DuckDB + summary/market/analysis/signal/portfolio/trade_plan/collector/briefing/scheduler API |
| 分析引擎 | 完整 | 6模块: technical/valuation/fund_flow/sector_rotation/signal/risk |
| 交易计划 | 完整 | CRUD + 评估 + 止盈止损监控 |
| 前端 | 完整 | React 18 + Dashboard / MarketSummary / StockAnalysis / SectorMap / Portfolio / TradePlan |
| 数据层 | 已落地真实路径 | `AKShareProvider` 已实现，`Tushare` 作为补强数据源接入 |
| 调度器 | V1已落地 | `APScheduler` 已接入，支持 market/news/daily_scan 任务 |
| 每日扫描与预警 | 已落地 | 扫描结果、预警落库，并在 Dashboard 展示 |
| B站字幕 | 独立工具 | yt-dlp + srt2md.py，已保留手动采集入口 |

### 1.2 当前仍存在的关键缺口

| 缺口 | 影响 | 严重度 |
|------|------|--------|
| 非结构化信息链路未闭环 | 新闻/社区/视频还未进入统一信号总线 | 高 |
| 无周线/月线MACD | 策略核心信号无法计算 | 高 |
| 无个股↔板块映射 | "高切低"在个股层面断裂 | 高 |
| 缺少正式 watchlist CRUD | 当前每日扫描主要依赖 `data/watchlist.json` | 高 |
| 无政策面权重 | 综合评分缺30%的权重 | 高 |
| AI 简报 / 调研层未落地 | 日报生成、研究项目、研究笔记仍停留在设计阶段 | 中 |

### 1.3 本轮已落地的新功能

- **每日扫描与预警**：新增 `DailyScanner`，扫描自选股、持仓股和固定核心标的；结果写入 `daily_scan_results`，预警写入 `alerts`。
- **Tushare 补强**：新增 `trade_cal` 与 `daily_info` 两条补强链路，分别落入 `trading_calendar` 与 `market_daily_stats`。
- **调度器 V1**：新增 `scheduler/engine.py` 与 `scheduler/jobs.py`，FastAPI lifespan 启动时自动注册并运行任务。
- **调度状态可视化**：新增 `/api/scheduler/status`、`/api/scheduler/history`，Dashboard 已展示调度器状态与最近运行历史。
- **扫描 API 与前端操作**：新增 `/api/briefing/scan/run`、`/api/briefing/scan/latest`、`/api/briefing/alerts`、`/api/briefing/alerts/{alert_id}/read`。

### 1.4 核心痛点

信息过载：数据源分散（行情/新闻/社区/视频），每天手动看太多内容，需要自动化 采集 → 筛选 → 分析 → 研究 的完整链路。

### 2.3 设计原则

1. **渐进增强** — 不重构V1，逐层叠加，每个Phase独立交付价值
2. **Provider模式延续** — 采集层各数据源沿用 `DataProvider` ABC 模式
3. **结构化优先** — 非结构化数据经LLM提取后变成结构化信号，复用现有引擎
4. **单人项目约束** — APScheduler单进程，DuckDB嵌入式，不引入重型基础设施
5. **DuckDB一库到底** — 所有数据存DuckDB，不引入额外数据库

## 三、采集层设计

### 3.1 AKShareProvider + Tushare 补强

当前版本已经以 `AKShareProvider` 作为主要真实数据路径，并额外接入 `TushareClient` 做日历与市场日度统计补强。

#### AKShare 主路径

`AKShareProvider` 已实现 `DataProvider` 现有方法与扩展方法，当前承担主要结构化行情输入：

| 方法 | AKShare接口 | 说明 |
|------|------------|------|
| `get_stock_daily()` | `ak.stock_zh_a_hist()` | 日K线 |
| `get_index_daily()` | `ak.stock_zh_index_daily()` | 指数日线 |
| `get_etf_flow()` | `ak.fund_etf_fund_daily_em()` | ETF资金流 |
| `get_margin_data()` | `ak.stock_margin_sse()` | 两融数据 |
| `get_northbound_flow()` | `ak.stock_hsgt_north_net_flow_in_em()` | 北向资金 |
| `get_stock_valuation()` | `ak.stock_a_indicator_lg()` | PE/PB/市值 |
| `get_sector_data()` | `ak.stock_board_industry_hist_em()` | 板块行情 |
| **新增** `get_stock_weekly()` | 日K聚合 | 周K线 |
| **新增** `get_stock_monthly()` | 日K聚合 | 月K线 |
| **新增** `get_sector_stocks()` | `ak.stock_board_industry_cons_em()` | 板块成分股 |
| **新增** `get_stock_sector()` | 反向映射 | 个股所属板块 |

当前 `tradepilot/config.py` 中 `DATA_PROVIDER = DataProviderType.AKSHARE`，默认开发配置已经不是纯 Mock。

**容错设计**：AKShare有请求频率限制，增加指数退避重试。严重限流时降级到Mock。

#### Tushare 补强路径（当前已实现）

| Tushare接口 | 落库表 | 用途 |
|------------|--------|------|
| `trade_cal` | `trading_calendar` | 交易日判断、调度任务非交易日跳过 |
| `daily_info` | `market_daily_stats` | 市场 breadth / 日度统计补强 |

当前实现约束：

- `TUSHARE_TOKEN` 通过仓库根目录 `.env` 读取。
- `daily_info` 按日拉取，并做了轻量节流。
- 手动大范围补数时增加 `_TUSHARE_MAX_SYNC_DAYS = 31` 限制，避免一次性打爆调用频率。

### 3.5 调度引擎

**文件**: `tradepilot/scheduler/engine.py`, `tradepilot/scheduler/jobs.py`

**技术选型**: APScheduler — 轻量级，单进程内运行，不需要外部依赖。

**当前版本状态**：已实现并接入 FastAPI lifespan，应用启动时自动 `start_scheduler()`，退出时 `stop_scheduler()`。

**当前已实现任务**:

| 任务 | 时间 | 说明 |
|------|------|------|
| 行情数据更新 | 周一至五 16:00 | 收盘后拉取当日行情 |
| 新闻采集 | 交易时段每30分钟 | 快讯+公告 |
| 每日扫描 | 周一至五 16:30 | 行情更新后运行 |

**当前运行细节**:

- `market_sync` 与 `daily_scan` 运行前会检查 `trade_cal`，非交易日直接记为 `skipped`。
- 每次执行都会写入 `scheduler_history`。
- 任务失败时会写 `scheduler_history`，并生成系统级 `alerts`。

**待扩展任务**:

- 社区采集
- 每日简报生成
- B站视频自动采集

**与FastAPI集成**:

```python
@asynccontextmanager
async def lifespan(app: FastAPI):
    start_scheduler()
    yield
    stop_scheduler()
```

**手动触发 / 状态查询**: 当前已暴露的 API 端点:

```
POST /api/collector/market/sync
POST /api/collector/news/sync
POST /api/briefing/scan/run

GET  /api/briefing/alerts

```

**当前接口可用性补充**:

- `POST /api/collector/market/sync`：当前可用，最近一轮实测可在短时间窗口内返回成功。
- `GET /api/signal/market_sentiment`：当前存在问题，暂不可用。真实数据路径会串行拉取北向资金、两融和 4 个 ETF 资金流，现阶段容易超时。
- `POST /api/briefing/scan/run`：当前存在问题，暂不可用。该接口依赖 `market_sentiment`，并且还会继续执行多标的扫描，当前同步链路无法稳定完成。
- TODO：为 `market_sentiment` 增加外部数据单次超时、缩短默认查询窗口，并评估缓存策略。
- TODO：将 `scan/run` 改为后台任务执行，返回 task/run id，再通过状态接口轮询结果。

## 四、筛选层设计

### 4.1 结构化筛选

**文件**: `tradepilot/filter/structured.py`

直接调用已有分析引擎，批量扫描自选股：

```python
class StructuredFilter:

```

### 4.2 非结构化筛选

**文件**: `tradepilot/filter/unstructured.py`

用LLM从新闻/社区/视频文本中提取标准化信号：

| 抽取字段 | 说明 |
|---------|------|
| stock_codes | 涉及的股票代码 |
| sector | 涉及的板块 |
| signal_type | policy / sentiment / fundamental / event |
| direction | bullish / bearish / neutral |
| strength | 1-5 |
| summary | 一句话总结(不超50字) |
| source_quote | 原文关键句(溯源) |

**成本控制**:
- 短文本(<200字): Claude Haiku
- 长文本(公告/研报/字幕): Claude Sonnet
- 批量合并: 10-20条短新闻一次请求
- 同一文本不重复提取

### 4.3 统一信号总线

**文件**: `tradepilot/filter/signal_bus.py`

所有来源的信号转为统一格式：

```python
class Signal(BaseModel):
    id: str                  # UUID
    timestamp: datetime
    source_type: str         # "structured" | "news" | "community" | "bilibili"
    source_id: str | None    # 原始数据ID(溯源)
    stock_codes: list[str]
    sector: str | None
    signal_type: str         # "technical" | "valuation" | "fund_flow" |
                             # "sector_rotation" | "policy" | "sentiment" |
                             # "fundamental" | "event"
    direction: str           # "bullish" | "bearish" | "neutral"
    strength: int            # 1-5
    summary: str
    details: dict
    expires_at: datetime | None
```

**综合评分权重(扩展后)**:

| 维度 | 权重 | 来源 |
|------|------|------|
| 政策面 | 30% | 新闻/公告 + 手动标注 |
| 资金面 | 25% | AKShare结构化 |
| 技术面 | 20% | AKShare结构化 |
| 估值面 | 15% | AKShare结构化 |
| 轮动 | 10% | AKShare + 新闻 |

非结构化信号按 `signal_type` 分配到对应权重类别。

### 5.3 主动预警

**文件**: `tradepilot/llm/alert.py`

| 预警类型 | 触发条件 | 紧急度 |
|---------|---------|--------|
| 止损警报 | 持仓触发任一止损条件 | 高 |
| 抽离提醒 | 情绪>=80 或 盈利>=30% | 高 |
| 建仓机会 | 自选股评分>=75 且 低位板块 | 中 |
| 高切低时机 | 持仓板块转高位 + 有低位替代 | 中 |
| 政策异动 | 新闻中strength>=4的政策信号 | 中 |
| 板块异动 | 板块单日涨幅>5%或跌幅>3% | 低 |

预警存入 `alerts` 表，前端 Dashboard 已展示最新预警。

**当前版本实际已落地的预警类型**:

- `stop_loss`
- `take_profit`
- `watchlist_opportunity`
- `core_signal`
- `rotation`
- `system`（调度或运行失败）

### 5.4 每日扫描引擎

**文件**: `tradepilot/scanner/daily.py`

核心价值模块，整合当前结构化分析结果与运行状态输出。

**当前版本已实现范围**:

1. 扫描 watchlist（当前主要来自 `data/watchlist.json`）
2. 扫描持仓股
3. 固定扫描核心标的（指数 / ETF）
4. 汇总板块轮动建议
5. 生成并落库扫描结果与预警
6. 读取最新 `market_daily_stats` 填充市场概览补强信息


## 十、技术决策

### 技术选型

| 决策 | 选择 | 理由 |
|------|------|------|
| 调度器 | APScheduler | 单进程，无需Redis/Celery |
| 数据库 | 继续DuckDB | 嵌入式无运维，分析性能好 |
| LLM | Claude API | 非结构化文本理解需要强推理 |
| HTTP客户端 | httpx | 异步，社区/新闻采集用 |
| 数据源 | AKShare + Tushare | AKShare 作为主路径，Tushare 只补交易日历与市场统计弱项 |

### 新增Nix依赖

```
akshare, apscheduler, anthropic, httpx, tushare
```

### 风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| AKShare接口不稳定 | 无法获取数据 | 本地缓存 + 增量更新 + Mock回退 |
| Claude API成本 | 月费超预算 | 短文本Haiku + 批量合并 + 日成本上限 |
| 社区反爬 | 社区数据采不到 | 社区非核心依赖，采集失败不影响主流程 |
| 单进程性能 | 扫描太慢 | DuckDB向量化查询本身很快，100只自选股秒级 |
